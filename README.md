# RefMind · 智能文献知识库助手

![RefMind Logo](https://img.shields.io/badge/RefMind-文献知识库助手-blue?style=for-the-badge&logo=readthedocs)

RefMind 是一个基于大语言模型的智能文献知识库助手，专为科研文献阅读场景设计。它致力于构建一个完整的端到端 RAG（检索增强生成）应用，旨在解决科研人员在处理海量文献时面临的效率低下、信息过载以及大模型“幻觉”等核心痛点。通过提供高精度 PDF 解析、混合检索 RAG 问答、多用户组独立知识库、长对话记忆、文档翻译与自动摘要等功能，RefMind 极大地提升了文献管理与知识获取的效率和准确性。

## ✨ 核心功能

RefMind 提供了一系列强大功能，助力用户高效管理和利用文献知识：

-   **高精度 PDF 解析**：优先调用 [MinerU](https://github.com/opendatalab/MinerU) 进行结构化解析，确保论文中的正文、公式、表格和排版结构得到准确提取。若 MinerU 未安装或解析失败，系统将智能回退到 PyMuPDF 进行文本抽取，保证解析的可用性。
-   **混合检索 RAG**：结合 BM25 关键词检索与 Chroma 向量检索，通过 `EnsembleRetriever` 以 0.5:0.5 的权重融合检索结果。支持中文 `jieba` 分词，兼顾语义泛化与关键词精确匹配，尤其适用于中英文混合的科研文献。
-   **多用户组隔离知识库**：每个用户组拥有独立的 Chroma 集合与持久化目录，实现文献、会话和消息的完全隔离管理，避免不同研究方向或用户之间的知识库混淆。
-   **智能长对话记忆**：系统保留最近 30 轮对话，并运用语义相似度过滤机制，仅将与当前问题高度相关的历史消息纳入上下文，有效降低 token 成本并提升问答准确性。
-   **文档翻译与自动摘要**：基于 `qwen3.7-plus` 等大模型，支持流式输出。摘要功能帮助用户快速了解文档核心内容；翻译功能则可结合当前文献库进行上下文增强，确保专业术语翻译的准确性。
-   **会话管理**：在同一文献库下，用户可以创建多个独立的对话会话，共享知识库但保持对话历史的独立性。

## 💡 项目定位与解决的问题

RefMind 的核心目标不是简单地“上传 PDF 后问答”，而是构建一个完整的端到端 RAG 应用，解决以下三个关键问题：

1.  **论文内容长、结构复杂，人工查找效率低**：RefMind 自动化了 PDF 解析、分块、向量化和入库过程，并提供智能问答，大幅提升了信息检索效率。
2.  **单纯使用大模型容易产生幻觉，需要严格基于文档回答**：通过严谨的 RAG 问答流程和“无检索结果则拒答”策略，RefMind 确保所有回答都严格基于用户上传的文献内容，并支持文档溯源。
3.  **不同研究方向或用户组的文献需要隔离管理**：多文献库隔离机制允许用户按需创建和管理独立的知识库，确保不同项目或研究领域的文献互不干扰。

## 🏗️ 整体架构设计

项目采用清晰的分层架构，将不同职责的模块组织到独立层中，提高了系统的可维护性和可扩展性。整体架构可分为六层：

1.  **配置层**：管理所有模型参数、API 地址、嵌入模型、解析方式、分块大小、检索 Top-K、记忆轮数等。通过 `.env` 文件进行配置，并支持在前端设置页动态修改，实现模型和参数的灵活切换，无需修改源码。
2.  **持久化层**：使用 SQLite 存储文献库、文档、会话和消息记录等结构化元数据。每个文献库可包含多篇文档和多个对话会话。Chroma 负责保存向量索引，与 SQLite 职责分离，共同实现数据持久化。
3.  **PDF 解析层**：优先使用 MinerU 进行高精度解析，以获取更丰富的结构化信息。若 MinerU 不可用或失败，则自动回退到 PyMuPDF 进行纯文本抽取。这种“高精度优先 + 可用性兜底”的设计确保了系统在不同环境下都能稳定运行。
4.  **RAG 核心层**：项目的技术核心，包括文档分块、向量化、Chroma 入库、BM25 检索、向量检索、混合检索以及基于 LangGraph 的问答流程。负责将原始文档转化为可检索知识，并进行智能问答。
5.  **业务编排层**：负责协调和串联上传 PDF 后的一系列流程，如保存原始文件、解析 PDF、生成分块、写入向量库、生成摘要、更新数据库状态、重建检索器缓存等。前端通过调用此层接口，降低了模块间的耦合度。
6.  **Streamlit 前端层**：提供用户交互界面，包括文献库管理、文件上传、解析进度展示、聊天问答、文档列表、翻译和设置页。前端主要承担交互和状态管理，核心业务逻辑则由后端模块处理。

## 🛠️ 技术栈

RefMind 项目基于以下核心技术构建：

-   **Python 3.11+**
-   **LangChain 1.0**：用于构建和管理 LLM 应用的核心框架。
-   **LangGraph 1.0**：用于构建有状态、多步骤的 LLM 应用程序，实现复杂的问答流程。
-   **Chroma**：轻量级向量数据库，用于存储和检索文档向量。
-   **BM25**：关键词检索算法，与向量检索结合实现混合检索。
-   **Streamlit**：用于快速构建交互式 Web 应用的前端框架。
-   **DashScope（OpenAI 兼容）**：阿里云提供的大模型服务，通过 OpenAI 兼容接口调用。

> ℹ️ **注意**：本项目基于 LangChain 1.0 架构。`EnsembleRetriever` 等遗留组件在 1.0 中已迁移至 `langchain-classic`，因此依赖中包含 `langchain-classic`。

## 🚀 快速开始

按照以下步骤，您可以快速部署并运行 RefMind 项目。

### 1. 环境准备

确保您的系统已安装 Python 3.11 或更高版本。

### 2. 克隆项目

```bash
git clone https://github.com/your-repo/RefMind.git # 替换为实际的项目仓库地址
cd RefMind
```

### 3. 创建并激活虚拟环境

```bash
python -m venv .venv
# Windows
.venv\Scripts\activate
# macOS / Linux
source .venv/bin/activate
```

### 4. 安装依赖

```bash
pip install -r requirements.txt
```

### 5. 配置环境变量

复制 `.env.example` 文件并重命名为 `.env`，然后编辑 `.env` 文件，填入您的 `DASHSCOPE_API_KEY`。

```bash
# Windows
copy .env.example .env
# macOS / Linux
cp .env.example .env
```

打开 `.env` 文件，您需要配置至少以下变量：

```ini
DASHSCOPE_API_KEY="YOUR_DASHSCOPE_API_KEY"
# API_BASE="https://dashscope.aliyuncs.com/compatible-openai/v1" # 默认值，如果需要可修改
# LLM_MODEL="qwen-plus" # 默认值，可根据需求修改
# EMBEDDING_MODEL="text-embedding-v2" # 默认值，可根据需求修改
```

### 6. 启动应用

```bash
streamlit run app.py
```

应用将在您的浏览器中自动打开，如果未跳转，请在浏览器中打开：http://localhost:8888/

### 7. 安装 MinerU（可选，用于高精度 PDF 解析）

RefMind 优先使用 MinerU 进行高精度 PDF 解析。如果您希望获得最佳解析效果，建议安装 MinerU。未安装时，项目将自动回退到 PyMuPDF 进行解析。

```bash
pip install mineru # 参见 MinerU 官方仓库获取完整安装说明：https://github.com/opendatalab/MinerU
```

安装 MinerU 后，请编辑 `.env` 文件，将 `MINERU_BINARY_PATH` 设置为对应的命令（通常是 `mineru` 或旧版 `magic-pdf`）：

```ini
MINERU_BINARY_PATH="mineru" # 或 "magic-pdf"
```

## 深入解析核心模块

### 📄 文档入库流程

用户上传 PDF 后，系统会执行一系列自动化步骤，将文档转化为可检索的知识：

1.  **保存原始 PDF**：上传文件被复制到本地数据目录，并记录到数据库，初始状态为 `parsing`。
2.  **解析 PDF**：优先使用 MinerU 提取正文、页码、Markdown 内容等结构化信息。若 MinerU 不可用或失败，则使用 PyMuPDF 按页抽取纯文本。所有解析结果统一为相同数据结构。
3.  **文档分块**：使用递归字符分割器，根据配置的 `chunk size` 和 `overlap` 将文档切分为适合检索和生成的片段。分块时保留文档名、页码、文献库 ID、chunk ID 等元数据，确保回答可溯源。
4.  **向量化入库**：每个分块通过 DashScope 的 OpenAI 兼容嵌入接口生成向量，并写入 Chroma。每个文献库拥有独立的 Chroma `collection` 和持久化目录，实现数据隔离。
5.  **生成摘要并更新状态**：系统选取文档前若干个分块，由大模型生成约 200 字的中文摘要，用于前端预览。最后文档状态变为 `ready`，并使当前文献库的检索器缓存失效，确保下次问答能检索到最新内容。

> ⚠️ **重复文档处理**：如果上传的 PDF 与当前文献库中已有同名文件，系统会先删除旧文档及其对应的向量分块，然后重新解析并入库，避免重复检索。

### 🔍 混合检索设计

RefMind 采用 BM25 + 向量检索的混合策略，以克服单一检索方式的局限性：

-   **向量检索**：擅长处理语义相近的问题，即使原文没有完全相同的词，也能找到语义相关片段。适用于概念性、抽象性问题的回答。
-   **BM25 关键词检索**：对精确词匹配更敏感，适用于论文中的专业术语、缩写、设备型号、算法名称等。项目使用 `jieba` 对中文文本进行分词，并保留英文和数字 token，以适应中英文混合内容。

最终，系统通过 LangChain 的 `EnsembleRetriever` 将 BM25 和 Chroma 向量检索结果融合，当前权重均为 0.5。这种设计兼顾了“语义泛化”和“关键词精确匹配”，在科研文献场景中表现更优。

> 🔄 **BM25 索引更新**：BM25 索引为内存索引。每次文档增删后，系统会主动让该文献库的检索器缓存失效，下次检索时重新构建 BM25 索引，确保检索结果与当前文献库内容一致。

### 💬 LangGraph 问答流程

问答部分使用 LangGraph 构建了一个简洁的状态机，核心流程为 `retrieve` 到 `generate`：

-   **`retrieve` 节点**：根据用户问题和当前文献库 ID 获取混合检索结果，专注于信息检索，不参与生成。
-   **`generate` 节点**：构造提示词，将用户问题、相关历史消息和检索到的参考文档交给大模型生成答案。**关键在于，如果没有检索到任何文档，它不会让大模型自由回答，而是直接返回固定话术：“已检索历史文档，暂未找到相关内容，无法回答。”**

这种“无检索结果则拒答”的策略，有效避免了大模型在无相关信息时产生“幻觉”，将回答范围严格限制在用户上传的论文内容内。系统提示词也明确要求模型只能基于参考文档回答，并尽量引用文档名和页码，实现了回答的可溯源能力。

### 🧠 长对话记忆机制

RefMind 实现了一个语义相关性过滤的长对话记忆机制，解决了传统滑动窗口记忆在论文问答场景中的局限性：

-   **问题**：在论文问答中，用户可能连续提问不同论文或不同主题的问题。将所有历史对话都塞入 prompt 会增加 token 成本，并可能干扰模型回答。
-   **解决方案**：系统从 SQLite 加载最近若干轮消息，将当前问题和历史消息分别进行 embedding，通过余弦相似度判断哪些历史消息与当前问题相关。只有相似度超过阈值的历史消息才会进入本轮问答上下文。

这种“按相关性记忆”的方式，既保留了多轮对话能力，又避免了无关历史信息的干扰，提高了问答的效率和准确性。

> ⚙️ **可用性兜底**：如果嵌入接口不可用，系统将自动回退到普通滑动窗口记忆，确保功能不会中断。

### 🔒 多文献库隔离

项目支持多个文献库，每个文献库都拥有独立的文档、会话、消息和向量库目录，实现数据的完全隔离：

-   **数据库层面**：文档、会话和消息均绑定到对应的文献库或会话。删除文献库时，会级联删除相关文档、会话、消息，并清理对应的 Chroma 向量目录。
-   **向量库层面**：每个文献库使用独立的 Chroma `collection` 和持久化目录。这确保了不同研究方向、不同用户或不同课题组的文献不会混淆，检索结果高度精准。

### 🌐 模型与 API 适配

RefMind 通过 DashScope 的 OpenAI 兼容接口调用阿里云大模型，并针对实际工程场景进行了优化：

1.  **懒加载与缓存**：模型客户端采用懒加载和缓存机制，仅在真正调用问答、翻译或嵌入时才创建客户端，减少启动失败概率。
2.  **批量大小限制**：显式配置 embedding batch size，避免一次提交过多文本导致 DashScope 嵌入接口报错。
3.  **动态配置**：前端设置页中修改 API Key、模型名、temperature、chunk size、retrieval top-k 等参数后，会写回 `.env` 文件，并清空模型缓存和检索器缓存，使新配置即时生效。

### 📝 翻译与摘要功能

除了核心 RAG 问答，RefMind 还提供了两个实用的辅助能力：

-   **自动摘要**：文档入库后，系统会选取前若干个文本分块，由大模型生成约 200 字的中文摘要，帮助用户在文档库页面快速了解论文核心内容。
-   **上下文增强翻译**：用户在翻译页输入文本后，系统会先从当前文献库检索相关片段，将这些片段作为术语参考传递给大模型，再进行翻译。这确保了翻译结果与当前文献库中的专业术语保持一致，提升了专业翻译的准确性。

这体现了项目中 RAG 能力的复用性：同一套检索系统不仅用于问答，也可以用于翻译增强。

## 🌟 关键技术点总结

RefMind 项目的核心技术亮点包括：

1.  **LangChain 1.0 与 LangGraph 的结合**：通过 LangGraph 将检索和生成拆分为独立的状态节点，使流程更清晰，并为未来扩展（如 query rewrite、rerank、工具调用等）奠定基础。
2.  **BM25 + 向量混合检索**：兼顾语义泛化与关键词精确匹配，尤其适用于中英文混合的科研文献场景。
3.  **语义过滤记忆**：解决长对话中的上下文噪声问题，根据当前问题筛选相关历史消息，而非简单保留最近 N 轮对话。
4.  **PDF 解析容错**：MinerU 提供高质量解析，PyMuPDF 提供 fallback 机制，兼顾解析效果与系统可用性。
5.  **多文献库数据隔离**：SQLite 元数据和 Chroma 向量库均按文献库隔离，适用于多课题、多用户或多项目场景。
6.  **工程化配置**：模型、检索、分块、记忆、解析等参数均可配置，并支持前端保存到 `.env`，提升了系统的可维护性和可迁移性。

## ⚙️ 配置项（.env）

项目的主要配置项通过 `.env` 文件管理，以下是常用配置及其说明：

| 变量名                       | 说明                                           | 示例值                                                |
| :--------------------------- | :--------------------------------------------- | :---------------------------------------------------- |
| `DASHSCOPE_API_KEY`          | 阿里云 DashScope API Key，用于调用大模型服务   | `sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`                |
| `API_BASE`                   | OpenAI 兼容接口地址，默认为 DashScope 兼容接口 | `https://dashscope.aliyuncs.com/compatible-openai/v1` |
| `LLM_MODEL`                  | 对话模型名称，例如 `qwen-plus`、`qwen-max` 等  | `qwen-plus`                                           |
| `EMBEDDING_MODEL`            | 嵌入模型名称，例如 `text-embedding-v2` 等      | `text-embedding-v2`                                   |
| `MINERU_BINARY_PATH`         | MinerU 命令路径，如已安装 MinerU 则配置此项    | `mineru` 或 `magic-pdf`                               |
| `USE_FALLBACK_PARSER`        | 强制使用 PyMuPDF 回退解析，即使 MinerU 已安装  | `False`                                               |
| `RETRIEVAL_TOP_K`            | 每个检索器（BM25 和向量检索）返回的片段数量    | `5`                                                   |
| `MEMORY_MAX_TURNS`           | 长对话记忆保留的最大轮数                       | `30`                                                  |
| `MEMORY_RELEVANCE_THRESHOLD` | 历史消息相关性阈值，用于语义过滤记忆           | `0.7`                                                 |
| `CHUNK_SIZE`                 | 文档分块大小（字符数）                         | `1000`                                                |
| `CHUNK_OVERLAP`              | 文档分块重叠大小（字符数）                     | `200`                                                 |

## 📝 注意事项
